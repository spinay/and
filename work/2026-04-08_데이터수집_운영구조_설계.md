# 공모주 알리미 — 데이터 수집/운영 구조 설계

> 작성일: 2026-04-08
> 전제: MVP 단계의 법적/운영 리스크 최소화. 관리자 승인형 구조.
> 스택: Node.js + PostgreSQL + React Native (모바일 클라이언트는 Flutter 유지 가능, API는 동일)

## 1. 전체 수집 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                        외부 소스                              │
│  [피너츠]   [38커뮤니케이션]   [DART OpenAPI]   [KRX 공시]   │
└──────┬──────────┬─────────────────┬────────────────┬────────┘
       │          │                 │                │
       ▼          ▼                 ▼                ▼
┌─────────────────────────┐  ┌──────────────────────────────┐
│ Crawler Workers         │  │ Verification Workers         │
│ (Playwright + cheerio)  │  │ (DART/KRX REST 호출)         │
│  - 폴라이트 fetch        │  │  - 같은 종목 키로 조회       │
│  - 1종목당 최소 필드     │  │  - 필드별 일치 비교          │
└────────────┬────────────┘  └────────────┬─────────────────┘
             │                            │
             ▼                            ▼
      ┌──────────────────────────────────────┐
      │ raw_snapshots (원본 JSON 보존)        │
      └────────────────┬─────────────────────┘
                       ▼
      ┌──────────────────────────────────────┐
      │ Normalizer (필드 표준화 + 계산)       │
      │  - 날짜/금액/주간사 정리              │
      │  - 최소 청약금 자동 계산               │
      └────────────────┬─────────────────────┘
                       ▼
      ┌──────────────────────────────────────┐
      │ Diff Engine                          │
      │  - 기존 published와 필드별 비교       │
      │  - 변경/신규/충돌 분류                │
      └────────────────┬─────────────────────┘
                       ▼
      ┌──────────────────────────────────────┐
      │ approval_queue (관리자 검토 대기)     │
      └────────────────┬─────────────────────┘
                       ▼ 승인
      ┌──────────────────────────────────────┐
      │ ipo_published (앱이 보는 공식 데이터)│
      └────────────────┬─────────────────────┘
                       ▼
      ┌──────────────────────────────────────┐
      │ Mobile API (Node/Express, 캐시)      │
      │ Push Worker (FCM/APNs)               │
      └──────────────────────────────────────┘
```

**원칙**
- 크롤링 결과는 절대 앱에 직접 노출되지 않음. 반드시 `approval_queue → ipo_published` 경로만 사용.
- 모든 단계가 idempotent. 같은 입력 → 같은 hash → 같은 결과.
- 실패는 단계별로 격리(재시도 가능). raw_snapshots는 영구 보존(추적·분쟁 대비).

**기술 스택**
- 스케줄러/큐: BullMQ + Redis
- 크롤러: Playwright (Node)
- API/Admin: Express + Prisma
- DB: PostgreSQL 15+
- Admin Web: Next.js (React)
- 모바일: React Native (또는 현 Flutter 유지 — 어느 쪽이든 API는 동일)

## 2. 데이터 모델

핵심 원칙: 수집 데이터(draft)와 게시 데이터(published)를 분리. 앱은 published만 본다.

```sql
-- 소스 메타
CREATE TABLE sources (
  id            SERIAL PRIMARY KEY,
  code          TEXT UNIQUE NOT NULL,    -- 'pnuts', '38', 'dart', 'krx'
  name          TEXT NOT NULL,
  kind          TEXT NOT NULL,           -- 'crawl' | 'official_api'
  base_url      TEXT,
  enabled       BOOLEAN DEFAULT true,
  rate_limit_ms INT DEFAULT 3000
);

-- 수집 원본 (불변, 추적용)
CREATE TABLE raw_snapshots (
  id            BIGSERIAL PRIMARY KEY,
  source_id     INT REFERENCES sources(id),
  external_key  TEXT,                    -- 소스 내 식별자(URL/페이지ID)
  fetched_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  http_status   INT,
  payload       JSONB NOT NULL,          -- 파싱한 원본
  payload_hash  TEXT NOT NULL,           -- 중복 fetch 방지
  UNIQUE(source_id, external_key, payload_hash)
);
CREATE INDEX ON raw_snapshots(source_id, fetched_at DESC);

-- 정규화된 후보(미게시)
CREATE TABLE ipo_drafts (
  id                    BIGSERIAL PRIMARY KEY,
  source_id             INT REFERENCES sources(id),
  raw_snapshot_id       BIGINT REFERENCES raw_snapshots(id),
  -- 핵심 식별
  ticker                TEXT,            -- 있으면 우선 매칭 키
  name_kr               TEXT NOT NULL,
  name_norm             TEXT NOT NULL,   -- 정규화 키 (공백/괄호 제거 등)
  -- 핵심 일정
  subscription_start    DATE,
  subscription_end      DATE,
  refund_date           DATE,
  listing_date          DATE,
  -- 가격
  price_band_low        INT,
  price_band_high       INT,
  final_price           INT,
  -- 청약
  underwriters          TEXT[],          -- 표준화된 증권사명 배열
  min_subscription_qty  INT,             -- 최소 청약 단위(보통 10)
  margin_rate           NUMERIC(4,3),    -- 증거금률 (0.500)
  -- 메타
  field_hashes          JSONB NOT NULL,  -- 필드별 해시 (diff용)
  record_hash           TEXT NOT NULL,
  created_at            TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX ON ipo_drafts(name_norm);
CREATE INDEX ON ipo_drafts(ticker);

-- 게시된 공식 데이터(앱이 읽는 테이블)
CREATE TABLE ipo_published (
  id                    BIGSERIAL PRIMARY KEY,
  ticker                TEXT,
  name_kr               TEXT NOT NULL,
  name_norm             TEXT NOT NULL UNIQUE,
  subscription_start    DATE,
  subscription_end      DATE,
  refund_date           DATE,
  listing_date          DATE,
  price_band_low        INT,
  price_band_high       INT,
  final_price           INT,
  underwriters          TEXT[],
  min_subscription_qty  INT,
  margin_rate           NUMERIC(4,3),
  -- 자동 계산 필드
  min_subscription_amount INT,           -- final_price * min_qty * margin_rate
  -- 메타
  field_hashes          JSONB NOT NULL,
  record_hash           TEXT NOT NULL,
  status                TEXT NOT NULL,   -- 'upcoming' | 'subscribing' | 'allocated' | 'listed' | 'closed'
  published_by          INT REFERENCES admin_users(id),
  published_at          TIMESTAMPTZ,
  updated_at            TIMESTAMPTZ DEFAULT now()
);

-- 승인 큐
CREATE TABLE approval_queue (
  id                BIGSERIAL PRIMARY KEY,
  draft_id          BIGINT REFERENCES ipo_drafts(id),
  target_published_id BIGINT REFERENCES ipo_published(id), -- null이면 신규 종목
  change_kind       TEXT NOT NULL,       -- 'create' | 'update' | 'conflict'
  diff              JSONB NOT NULL,      -- [{field, old, new, source}, ...]
  verification      JSONB,               -- DART/KRX 비교 결과 (있으면)
  status            TEXT NOT NULL DEFAULT 'pending', -- pending|approved|rejected|stale
  reviewed_by       INT REFERENCES admin_users(id),
  reviewed_at       TIMESTAMPTZ,
  reject_reason     TEXT,
  created_at        TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX ON approval_queue(status, created_at);

-- 변경 이력 (감사용)
CREATE TABLE ipo_change_logs (
  id                BIGSERIAL PRIMARY KEY,
  published_id      BIGINT REFERENCES ipo_published(id),
  field             TEXT NOT NULL,
  old_value         JSONB,
  new_value         JSONB,
  source_id         INT REFERENCES sources(id),
  approved_by       INT REFERENCES admin_users(id),
  changed_at        TIMESTAMPTZ DEFAULT now()
);

-- 검증 결과 (DART/KRX 대조)
CREATE TABLE verification_results (
  id              BIGSERIAL PRIMARY KEY,
  draft_id        BIGINT REFERENCES ipo_drafts(id),
  source_id       INT REFERENCES sources(id), -- DART/KRX
  field           TEXT NOT NULL,
  draft_value     JSONB,
  official_value  JSONB,
  match           BOOLEAN NOT NULL,
  checked_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE admin_users (
  id        SERIAL PRIMARY KEY,
  email     TEXT UNIQUE NOT NULL,
  role      TEXT NOT NULL,             -- 'editor' | 'admin'
  created_at TIMESTAMPTZ DEFAULT now()
);
```

필드 최소화: 위 컬럼이 곧 "수집해도 되는 전부". 그 외 정보(기업 소개, 업종 설명 등)는 관리자가 손으로 작성하거나 DART에서 검증된 텍스트만 사용.

## 3. 관리자 CMS 화면 설계

Next.js로 별도 도메인(`admin.ipokeeper.com`). 5개 화면이면 충분.

### 3-1. 대시보드 (`/`)
- 오늘 수집 건수 / 승인 대기 / 충돌 건수 카드
- 내일 청약 시작 종목 미리보기 (게시된 데이터 기준)
- 소스별 마지막 수집 시각 + 실패 알림

### 3-2. 승인 큐 (`/approvals`)
가장 중요한 화면. 좌우 분할:

```
┌──────────────────┬───────────────────────────┐
│ 대기 목록         │ 상세 (선택된 항목)         │
│                  │                            │
│ • 케이엔에스      │ ┌─ 변경 사항 ─────────┐   │
│   [update]       │ │ 청약마감 11/4 → 11/5 │   │
│   diff 2건       │ │ 공모가 22,000 (신규) │   │
│                  │ └──────────────────────┘   │
│ • XX바이오        │                            │
│   [create] 신규   │ ┌─ 검증 ─────────────┐   │
│                  │ │ DART  ✓ 일치        │   │
│ • OO반도체       │ │ KRX   — (미상장)    │   │
│   [conflict]     │ └──────────────────────┘   │
│   diff 5건       │                            │
│                  │ ┌─ 원본 출처 ────────┐   │
│                  │ │ 피너츠 raw 보기 →   │   │
│                  │ └──────────────────────┘   │
│                  │                            │
│                  │ [✓ 승인]  [✕ 반려]         │
└──────────────────┴───────────────────────────┘
```

- diff는 필드별 색상 표시(추가=초록, 변경=노랑, 삭제=빨강)
- 검증 섹션이 모두 ✓이면 "원클릭 승인" 단축키 활성
- 반려 시 사유 필수, 같은 raw_snapshot은 24시간 동안 재제안 안 됨

### 3-3. 종목 직접 편집 (`/ipos/:id`)
- 모든 필드 수동 편집 폼 (크롤러가 못 잡은 경우, 또는 정정 시)
- 저장 시 published에 직접 반영 + change_log 기록

### 3-4. 수집 로그 (`/runs`)
- 소스별 실행 이력, HTTP 상태, 파싱 실패, 처리 시간
- 실패한 raw_snapshot 재처리 버튼

### 3-5. 소스 관리 (`/sources`)
- 활성/비활성 토글, rate limit 조정, 셀렉터 버전 관리

## 4. 크롤링 파이프라인

```
[BullMQ scheduler]
  └─ "fetch:pnuts:list" job (cron: 매일 09:00)
       │
       ▼
[FetchWorker] ── HTML 받아서 → raw_snapshots에 INSERT
       │ (payload_hash 중복이면 skip하고 종료)
       ▼
"normalize:<raw_id>" job 생성
       │
       ▼
[NormalizeWorker]
  - parse → 정규화 → ipo_drafts INSERT
  - 같은 name_norm의 published 조회
       │
       ▼
"verify:<draft_id>" + "diff:<draft_id>" job 동시 생성
       │
       ▼
[VerifyWorker]                [DiffWorker]
  DART/KRX 호출                 기존 published와 비교
  verification_results 채움      변경 없으면 종료
       │                        변경 있으면 approval_queue에 INSERT
       └────────────┬───────────┘
                    ▼
            관리자 검토 (CMS)
```

**잡 단위 설계 원칙**
- 한 잡 = 한 raw_snapshot 단위. 절대 "전체 다시 돌리기" 없음.
- 모든 잡은 `attempts: 3, backoff: exponential`
- 잡 결과는 항상 다음 잡을 enqueue로만 연결 (직접 함수 호출 금지 → 부분 실패 시 추적 가능)

**Polite 크롤링 규칙**
- User-Agent 명시 (`IPOKeeperBot/1.0 contact@…`)
- 소스당 동시 1개, 요청 간격 ≥3초
- 새벽~오전 시간대에만 실행
- 실패 시 즉시 재시도 안 함 (15분 후 재시도)
- robots.txt 존중. 거부 페이지는 영구 스킵 리스트

## 5. 정규화 로직

```ts
// normalize/name.ts
export function normalizeName(raw: string): string {
  return raw
    .replace(/\([^)]*\)/g, '')   // (주), (스팩) 등 괄호 제거
    .replace(/\s+/g, '')
    .replace(/[·ㆍ・]/g, '')
    .toUpperCase();
}

// normalize/date.ts — "2026.11.04 ~ 11.05" 같은 케이스 처리
export function parseKoreanDateRange(s: string, year = new Date().getFullYear()) {
  const m = s.match(/(\d{4})?\.?(\d{1,2})\.(\d{1,2})\s*[~∼-]\s*(?:(\d{4})\.)?(?:(\d{1,2})\.)?(\d{1,2})/);
  if (!m) return null;
  const y1 = +(m[1] ?? year);
  const start = new Date(y1, +m[2] - 1, +m[3]);
  const y2 = +(m[4] ?? y1);
  const mm2 = +(m[5] ?? m[2]);
  const end = new Date(y2, mm2 - 1, +m[6]);
  return { start, end };
}

// normalize/price.ts — "22,000원", "20,000~25,000" 등
export function parsePriceBand(s: string): { low?: number; high?: number; final?: number } {
  const nums = [...s.matchAll(/[\d,]+/g)].map(x => +x[0].replace(/,/g, ''));
  if (nums.length === 1) return { final: nums[0] };
  if (nums.length >= 2) return { low: Math.min(...nums), high: Math.max(...nums) };
  return {};
}

// normalize/underwriter.ts — 표준 사전 매핑
const UW_MAP: Record<string, string> = {
  '한국투자': '한국투자증권',
  '한투': '한국투자증권',
  '미래에셋': '미래에셋증권',
  'NH': 'NH투자증권',
  'KB': 'KB증권',
  // …
};
export function normalizeUnderwriters(raw: string): string[] {
  return raw
    .split(/[,/、]| 및 /)
    .map(s => s.trim())
    .filter(Boolean)
    .map(s => UW_MAP[s] ?? (s.endsWith('증권') ? s : `${s}증권`));
}

// calc/min_amount.ts
export function calcMinSubscriptionAmount(price: number, qty = 10, marginRate = 0.5) {
  return Math.ceil(price * qty * marginRate);
}
```

원칙: 정규화 함수는 모두 순수 함수. 입력만 보면 출력 예측 가능. 단위 테스트로 케이스 누적 (실제 크롤링에서 만난 진짜 문자열을 픽스처로 보관).

## 6. 변경 감지 로직

필드 단위 해시로 비교. JSON diff 라이브러리 안 씀(필드 의미가 명확하므로 단순 비교가 더 안전).

```ts
// diff/hash.ts
import crypto from 'crypto';

const FIELDS = [
  'subscription_start', 'subscription_end', 'refund_date', 'listing_date',
  'price_band_low', 'price_band_high', 'final_price',
  'underwriters', 'min_subscription_qty', 'margin_rate',
] as const;

export function computeFieldHashes(rec: Record<string, any>) {
  const out: Record<string, string> = {};
  for (const f of FIELDS) {
    const v = rec[f];
    out[f] = crypto.createHash('sha1')
      .update(JSON.stringify(v ?? null))
      .digest('hex').slice(0, 12);
  }
  return out;
}

export function computeRecordHash(fieldHashes: Record<string, string>) {
  return crypto.createHash('sha1')
    .update(Object.entries(fieldHashes).sort().map(([k, v]) => `${k}:${v}`).join('|'))
    .digest('hex').slice(0, 16);
}

// diff/diff.ts
export function buildDiff(draft: any, published: any | null, sourceCode: string) {
  if (!published) {
    return { changeKind: 'create' as const, diff: FIELDS
      .filter(f => draft[f] != null)
      .map(f => ({ field: f, old: null, new: draft[f], source: sourceCode })) };
  }
  const changes: any[] = [];
  for (const f of FIELDS) {
    if (draft.field_hashes[f] !== published.field_hashes[f]) {
      changes.push({ field: f, old: published[f], new: draft[f], source: sourceCode });
    }
  }
  if (changes.length === 0) return null; // no-op
  return { changeKind: 'update' as const, diff: changes };
}
```

**충돌 감지**: 같은 종목이 서로 다른 소스에서 같은 시간대에 다른 값을 제출하면 `change_kind = 'conflict'`로 표시. 관리자가 양쪽을 보고 선택.

**stale 처리**: approval_queue 항목이 24시간 이상 미검토 상태인데 그 사이에 새로운 raw_snapshot이 들어오면 기존 큐는 `stale`로 닫고 새로 만든다.

## 7. Playwright 기반 예시 코드

```ts
// crawlers/pnuts.ts
import { chromium, Browser } from 'playwright';
import crypto from 'crypto';
import { db } from '../db';
import { normalizeName, parseKoreanDateRange, parsePriceBand, normalizeUnderwriters } from '../normalize';

const SOURCE_CODE = 'pnuts';
const LIST_URL = 'https://example-pnuts.example/ipo/list';
const USER_AGENT = 'IPOKeeperBot/1.0 (+contact@ipokeeper.app)';

export async function runPnutsCrawl() {
  const source = await db.source.findUniqueOrThrow({ where: { code: SOURCE_CODE } });
  if (!source.enabled) return;

  const browser: Browser = await chromium.launch({ headless: true });
  const ctx = await browser.newContext({ userAgent: USER_AGENT, locale: 'ko-KR' });
  const page = await ctx.newPage();

  try {
    await page.goto(LIST_URL, { waitUntil: 'domcontentloaded', timeout: 30_000 });
    await page.waitForSelector('table.ipo-list tr', { timeout: 10_000 });

    const rows = await page.$$eval('table.ipo-list tr.ipo-row', (trs) =>
      trs.map((tr) => ({
        name:        tr.querySelector('.col-name')?.textContent?.trim() ?? '',
        sub_period:  tr.querySelector('.col-sub')?.textContent?.trim() ?? '',
        refund:      tr.querySelector('.col-refund')?.textContent?.trim() ?? '',
        listing:     tr.querySelector('.col-listing')?.textContent?.trim() ?? '',
        price:       tr.querySelector('.col-price')?.textContent?.trim() ?? '',
        underwriter: tr.querySelector('.col-uw')?.textContent?.trim() ?? '',
        link:        (tr.querySelector('.col-name a') as HTMLAnchorElement | null)?.href ?? '',
      })),
    );

    for (const r of rows) {
      if (!r.name) continue;

      const payload = r;
      const payloadHash = crypto
        .createHash('sha1').update(JSON.stringify(payload)).digest('hex');

      const raw = await db.rawSnapshot.upsert({
        where: {
          source_id_external_key_payload_hash: {
            source_id: source.id,
            external_key: r.link || r.name,
            payload_hash: payloadHash,
          },
        },
        create: {
          source_id: source.id,
          external_key: r.link || r.name,
          payload,
          payload_hash: payloadHash,
          http_status: 200,
        },
        update: {},
      });

      const sub = parseKoreanDateRange(r.sub_period);
      const refund = parseKoreanDateRange(r.refund);
      const listing = parseKoreanDateRange(r.listing);
      const price = parsePriceBand(r.price);
      const underwriters = normalizeUnderwriters(r.underwriter);

      await enqueueNormalizeJob(raw.id, {
        name_kr: r.name,
        name_norm: normalizeName(r.name),
        subscription_start: sub?.start,
        subscription_end: sub?.end,
        refund_date: refund?.start,
        listing_date: listing?.start,
        price_band_low: price.low,
        price_band_high: price.high,
        final_price: price.final,
        underwriters,
        min_subscription_qty: 10,
        margin_rate: 0.5,
      });

      await page.waitForTimeout(source.rate_limit_ms);
    }
  } finally {
    await ctx.close();
    await browser.close();
  }
}
```

```ts
// workers/normalize.worker.ts
import { Worker } from 'bullmq';
import { db } from '../db';
import { computeFieldHashes, computeRecordHash, buildDiff } from '../diff';

new Worker('normalize', async (job) => {
  const { rawId, fields } = job.data;
  const fieldHashes = computeFieldHashes(fields);
  const recordHash = computeRecordHash(fieldHashes);

  const draft = await db.ipoDraft.create({
    data: { ...fields, raw_snapshot_id: rawId, field_hashes: fieldHashes, record_hash: recordHash },
  });

  const published = await db.ipoPublished.findUnique({ where: { name_norm: fields.name_norm } });
  const diff = buildDiff(
    { ...fields, field_hashes: fieldHashes },
    published,
    'pnuts',
  );
  if (!diff) return;

  await db.approvalQueue.create({
    data: {
      draft_id: draft.id,
      target_published_id: published?.id ?? null,
      change_kind: diff.changeKind,
      diff: diff.diff,
    },
  });

  await job.queue.add('verify', { draftId: draft.id });
});
```

## 8. DART/KRX를 보조 검증 소스로 섞는 방식

**역할 구분**
- 피너츠/38 = 속도/커버리지 (먼저 들어옴, 비공식)
- DART/KRX = 신뢰성 (느리지만 공식)

크롤링 데이터를 대체하는 게 아니라 대조한다. 일치하면 관리자 승인이 단순해지고, 불일치하면 경고를 띄운다.

### DART OpenAPI
- 엔드포인트: 공시검색(`list.json`) → 보고서명 LIKE `%증권신고서%` → `document.json`으로 본문 XML 받기
- 추출 대상 필드: 최종 공모가, 청약 일정, 대표주관회사
- 매칭 키: 회사명 정규화 + (있으면) 사업자등록번호. 텍스트 유사도 ≥ 0.9 요구.

### KRX
- 신규상장 일정/완료 페이지
- 추출 대상: 종목코드, 상장일 (이 두 개가 가장 가치 있음)

### 검증 워커 (의사코드)
```ts
async function verify(draftId: number) {
  const draft = await db.ipoDraft.findUniqueOrThrow({ where: { id: draftId } });

  const dart = await fetchDartReport(draft.name_kr);
  const krx  = await fetchKrxListing(draft.name_kr);

  const checks = [
    { source: 'dart', field: 'final_price',   official: dart?.finalPrice },
    { source: 'dart', field: 'subscription_start', official: dart?.subStart },
    { source: 'dart', field: 'underwriters',  official: dart?.underwriters },
    { source: 'krx',  field: 'listing_date',  official: krx?.listingDate },
    { source: 'krx',  field: 'ticker',        official: krx?.ticker },
  ];

  for (const c of checks) {
    if (c.official == null) continue;
    const draftVal = (draft as any)[c.field];
    const match = JSON.stringify(draftVal) === JSON.stringify(c.official);
    await db.verificationResult.create({
      data: { draft_id: draft.id, source_id: SRC[c.source], field: c.field,
              draft_value: draftVal, official_value: c.official, match },
    });
  }
}
```

승인 큐 상세 화면에 검증 결과 요약(✓/✗/—)이 나와 관리자가 즉시 판단 가능. 검증이 모두 ✓이면 "안전 승인" 표시.

중요: DART/KRX도 관리자 승인 없이 자동 게시되지 않음(MVP). `auto_approvable` 플래그로 관리자가 빠르게 통과시킬 수 있게만 함.

## 9. MVP / 확장 단계별 운영 전략

| 단계 | 데이터 흐름 | 관리자 부담 | 자동화 | 리스크 |
|------|-------------|------------|--------|--------|
| Phase 0 — 수동 시드 | 관리자가 직접 입력 | 매일 30분 | 없음 | 거의 없음 |
| Phase 1 — 반자동 (MVP 출시) | 크롤 → 정규화 → 승인 큐 → 클릭 승인 | 매일 5~10분 | 정규화/diff/검증 자동 | 낮음 |
| Phase 2 — 검증 자동 게시 | DART/KRX 모두 ✓ 인 항목은 자동 게시 | 예외만 처리 | 게시까지 자동 | 중간 |
| Phase 3 — 사용자 제보 통합 | 사용자 제보 → 신뢰도 점수 → 별도 큐 | 제보 검수 추가 | 점수 기반 우선순위 | 모더레이션 부담 |

### Phase 0 (출시 전 1~2주)
- crawler/admin은 모두 떠 있되, 모든 항목이 큐에 들어오고 관리자가 100% 승인
- 목적: 정규화 함수의 엣지케이스 수집
- 산출물: 정규화 단위 테스트 케이스, 셀렉터 안정성 검증

### Phase 1 (MVP 출시)
- 100% 관리자 승인 유지. 자동 게시 없음.
- 앱에는 published만 노출. 데이터 출처는 표시하지 않음 (책임 분리, 법적 리스크 ↓)
- 푸시 발송 주체는 앱(서버 푸시 워커), 트리거는 published 데이터의 날짜만.
- 운영 SLA: "청약 시작 12시간 전까지 published 반영" 정도면 충분.

### Phase 2 (3~6개월 후)
- 자동 승인 조건:
  1. 동일 종목이 서로 다른 두 크롤 소스에서 같은 값으로 들어옴
  2. DART 또는 KRX 검증 ✓
  3. 변경 필드가 위험 필드(가격·일정)가 아님
- 위험 필드 화이트리스트로 자동/수동 분기.

### Phase 3 (사용자 기반이 생긴 후)
- 사용자 제보 채널 (이미 청약한 사람의 정정 제보)
- 제보 → 별도 `user_reports` 테이블 → 관리자 확인 후 published 반영
- Slack/Discord 알림으로 관리자에게 즉시 통지

### 운영 사고 방지 원칙 (전 단계 공통)
1. published는 단일 진실 소스. 앱은 다른 어디도 보지 않음.
2. 모든 변경은 change_logs에 남는다. 누가/언제/무엇을/어디서.
3. 롤백 1클릭: 마지막 change_log 하나를 되돌리는 버튼.
4. 푸시는 반드시 published 시점 + 1시간 지연. 막 게시된 데이터에 오류 있으면 푸시 전에 정정 가능.
5. 원본 보존: raw_snapshots는 절대 삭제 안 함.

## 핵심 요약

이 설계의 핵심은 **"수집은 자동, 게시는 수동"**.
크롤링 결과가 앱으로 직접 가지 않는 한 법적/품질 리스크는 거의 모두 차단되고, 그 사이에 DART/KRX 검증으로 데이터 정확도를 점진적으로 높일 수 있음.
